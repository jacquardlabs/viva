| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | Trigger stays a soft judgment call and the fresh-context executor — the exact #113 persona — misses the new branch, runs the plain clear-state block, and deletes `review-input-rN.json`/`review-rN.json` before copying them. The bug the story exists to fix silently recurs. | At build, confirm SKILL.md's new branch leads with a concrete, deterministic gate (e.g. `grep -q '^## Revision History$' <doc>`), not prose that only says "recognize from context." At acceptance, dogfood a resume and confirm the second session reports `K > 0` pre-approved. |
| 2 | technical | The build picks preserved-copy names that a destructive glob still matches — either clear-state's `.viva/review-input-r*.json` / `review-r*.json` (deletes the copies before round 1 reads them), or `revision_history.py`'s `review-input-r*` sign-off glob (double-counts the prior session's notes into the new ledger block). | Grep the final SKILL.md for the chosen names; confirm neither contains the substring `review-input-r` or `review-r` immediately after `.viva/`. Both globs then provably miss them. |
| 3 | technical | Step 5 (discard the two copies after round-1 parse succeeds) is omitted or mis-ordered, so a *third* resume reads a two-sessions-old pair and carries forward stale approvals from session A over session B's real state. | Confirm SKILL.md deletes `prior-review-input.json` / `prior-review-verdicts.json` only after `review-input-r1.json` is written, and chains the discard to that parse's success (`&&`), not unconditionally or before it. |
| 4 | product | A doc carries a hand-written `## Revision History` heading (a changelog section) but was never reviewed by viva, so no `.viva/review-*` pair exists. The heading-based trigger fires but there is nothing to preserve; if step 1's "if neither file present, fall through" is dropped, the branch errors or references a nonexistent pair instead of degrading to the default first-review path. | Confirm the written procedure runs the "neither file present → fall through to default launch" check *before* any copy/carry-forward step; verify it no-ops cleanly when the heading exists but no round pair does. |
| 5 | product | `.viva/` is flat and un-namespaced. A leftover `review-input-r*.json` / `review-r*.json` pair from an *unrelated* doc's abandoned session is the highest-numbered pair present, so the resume branch copies it and feeds a different doc's approvals into this doc's round 1. | `_load_approved` gates carry-forward on `schema.section_key(title)` + byte-identical content, so mismatched sections cannot carry — confirm at acceptance that a cross-doc leftover yields `K = 0`, not silent wrong approvals. Verify the written trigger ties detection to the doc's own heading (doc-specific), the reason "no auto-detect from disk state alone" was chosen. |

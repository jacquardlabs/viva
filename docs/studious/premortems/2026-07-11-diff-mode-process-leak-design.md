| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | The empty-diff mid-loop exit is not the dominant cause of the 5 observed orphans. The diagnosis scopes out terminal-close abandonment as "not diff-mode-specific," but a persona who mostly runs `/viva-diff` would produce diff-mode orphans by abandoning sessions — a path this fix leaves untouched — so orphans keep accumulating after the fix ships. | After merge, watch whether new `--mode diff` orphans still appear in dogfood. Confirm the repro (a `changes` comment on the last remaining hunk that fully reverts it) is a path the persona actually hits with any frequency, versus a rare shape chosen because it is the one this fix closes. This is the Open-questions signal the doc itself names for opening a follow-up. |
| 2 | product | The reused step-5 sign-off template "N hunks approved across M files in R round(s)" misdescribes the empty-diff finish: at least one hunk was *reverted* on the human's request, not approved, so the report tells the human they approved a change they explicitly asked to drop — a fidelity gap against principle 2 ("Verbatim, not summarized"). | At build/acceptance, read the exact string the empty-diff branch emits. It should distinguish "diff fully resolved" from "all hunks approved," and the counters (esp. "K hunks revised") should account for the reverted hunk rather than counting it as approved. |
| 3 | technical | The browser tab is still showing the last hunk as `pending` when `/complete` fires from step 4. The SSE `complete` handler is generic across modes, but the combination — completion overlay rendered over an un-approved pending card — is untested and could show the human a confusing half-state instead of a clean sign-off. | Build-phase: drive the empty-diff path against a real browser (or assert the `complete` SSE render path) and confirm the tab shows an unambiguous sign-off, not "complete" layered over a pending hunk. |
| 4 | technical | The `N`/`M`/`K` counters the new call site sends may not be assembled at step 4. The doc claims the agent "already tracks" them, but the empty-diff branch historically only printed and exited — it never built a `/complete` payload before. The server tolerates malformed JSON (`json.loads` failure → `{}` at `server.py:3176`), so a wrong or zeroed payload shuts the server down but silently corrupts the sign-off report the human reads. | Review the new call site: confirm `N`/`M`/`K` are genuinely in scope and correct at diff.md step 4, not defaulted/zeroed. Assert the report values, not just that shutdown happened. |
| 5 | product | This fix closes one door, not the class. The launch guard (`diff.md:36`, hard `exit 1` when `.viva/server.url` exists) still turns *any* other abnormal exit — a crash, a killed agent, either scoped-out cause — into a hard block on the persona's next `/viva-diff` in that repo. The reported symptom (next session refused) can recur from a path this fix does not touch. | After merge, check whether the persona can still reach "a prior session may still be running" from any non-empty-diff abnormal exit. If the guard remains the single brittle chokepoint, weigh whether the follow-up should harden the guard (stale-PID detection) rather than add per-path `/complete` calls. |
| 6 | technical | The proposed companion test re-covers the primitive, not the new control flow. Operational readiness proposes POSTing the same `/complete` payload and asserting shutdown — but that re-tests `test_diff_mode_complete_shuts_down`'s already-covered server behavior, not that diff.md's `if [ ! -s .viva/diff.patch ]` branch actually reaches `/complete`. Because the skill is agent-executed prose (no `main()`), nothing mechanically verifies the branch fires. | At acceptance, confirm the added coverage exercises the empty-diff *condition* reaching the call (e.g. a fixture where a round-2 re-diff is empty), not merely the endpoint in isolation — otherwise a regression that drops the new call site passes CI green. |

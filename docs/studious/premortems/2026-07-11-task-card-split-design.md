| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | The `--split-on` path is threaded *through* `_find_split_level`/`_split_sections` instead of being a sibling branch that calls them unchanged, so "default parsing is byte-for-byte unchanged" becomes a claim, not a structural fact. | Diff `scripts/parse_sections.py`: the existing bodies of `_find_split_level` and the level-selection block of `_split_sections` should be untouched; the pattern rule should live at a new earlier decision point. Every existing assertion in `tests/test_parse_sections.py` must pass unmodified. |
| 2 | technical | A second identity rule sneaks in for task cards (epic pre-mortem item 4): the pattern path transforms the matched heading (strips a `Task N:` prefix, lowercases, trims differently) before it lands in `sections[].title`, so `section_key()` mismatches round-to-round for this path only and approval/annotation/diff/open-note carry-forward silently breaks. | Confirm the raw matched heading title is fed into `title` with no extra transform. Add a round-2 reparse test on a `--split-on` fixture asserting an approved task carries forward (`approved_ids` non-empty) and that `_carry_annotations`/`_compute_diffs`/`_attach_open_notes` still key correctly. |
| 3 | technical | Zero matches falls back silently instead of erroring. An implementer lets a non-matching `PATTERN` drop through to auto-detection or the whole-doc single section, hiding a caller's wrong pattern as a one-card review of the entire plan. | Run `parse_sections.py <fixture> --split-on '^NoSuchHeading'`: must exit non-zero naming the pattern and the doc, not write a one-section JSON. |
| 4 | product | The Revision-History swallow (design's Open question #1) bites jig in practice: after sign-off appends `## Revision History`, a re-review with `--split-on '^Task \d+'` swallows the ledger block into the last task's `content` and shows it to the reviewer as task body. | Parse a `PLAN.md` that already carries an appended `## Revision History` with `--split-on '^Task \d+'`; check whether the ledger text lands inside the final section's `content` rather than being excluded. Deliberately scoped out this story — verify it did NOT get "fixed" in a way that touched the default path. |
| 5 | product | The documentation deliverable silently doesn't ship. The design makes the `parse_sections.py` docstring, `SKILL.md` parsing-rules bullets, and README "Server CLI (advanced)" updates part of "done," but docs are the first thing dropped under implementation pressure, leaving jig integrators without the flag's contract. | Grep for `--split-on` in `scripts/parse_sections.py`'s module docstring, `SKILL.md`, and `README`; each must name the flag, its matching rule, and the "no `--split-on` → unchanged" guarantee. |
| 6 | technical | The acceptance-criteria fixture `PLAN.md` is missing, or added but only count-asserted. The story requires a fixture that parses "to one section per task with no custom parsing needed downstream" — that means a carry-forward round-trip assertion, not just counting cards. | A new fixture with `### Task N` blocks under `tests/`, plus a test asserting one section per task AND round-2 approval carry-forward through `section_key()` — not merely `len(sections) == N`. |
| 7 | technical | A malformed `--split-on` regex crashes with a raw `re.error` traceback instead of a clean caller-facing message. The design specifies zero-match handling but not invalid-pattern handling; a bad pattern from a call site should fail loud but legibly. | Run `--split-on '('` (unbalanced group): expect a one-line error naming the bad pattern, not a Python traceback. |

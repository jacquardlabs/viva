| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | product | The "recommended" badge reads to the reviewing human as a default or required pick, nudging a rubber-stamp of the agent's choice — eroding Principle 3's "the human alone decides." | Acceptance walkthrough of a card with `recommended_choice` set: is the recommended chip pre-selected, focus-defaulted, or styled as the primary action? Inspect `buildQACard` (server.py ~2364) for any auto-select/default-focus on the badged chip. |
| 2 | product | With a badge drawing the eye to one chip, the human acts on the badge without reading the recommendation's "why" (which stays in `hint`/choice text per Out of scope) — the fork decision degrades to "click the marked one." | Acceptance: with `recommended_choice` set, is the `hint` still rendered and prominent, or does the badge visually outrank the reason? Check the hint `<p class="section-summary">` still renders alongside the badged chip. |
| 3 | technical | Value-based matching misfires: a `recommended_choice` that doesn't exactly equal a `choices` entry (typo, trailing space, case) badges nothing or the wrong chip, silently — the exact failure the design rejected index-based matching to avoid. | Check `validate_qa_input` (scripts/schema.py) rejects a non-matching value with `ValueError`, and the render's chip match is exact string equality, not trimmed/fuzzy. Regression test with a near-miss value. |
| 4 | technical | Validation ships weaker than designed: the doc leaves hard-fail-vs-warning as an open question, so a non-fatal warning or omitted check lets a dangling `recommended_choice` degrade to a silent no-badge misfire at runtime. | Confirm `validate_qa_input` has a `recommended_choice` branch that raises `ValueError` at startup (matching existing failure modes), backed by a regression test — not a `print`/log-and-continue. |
| 5 | technical | `choices`-absent path crashes: when `recommended_choice` is present but `choices` is missing, a validator that tests membership before guarding presence throws on `None`/missing instead of the clean `ValueError` the design specifies. | Test `validate_qa_input` with `recommended_choice` set and no `choices` key — expect a descriptive `ValueError`, not a `TypeError`/`KeyError`. Verify the presence guard precedes the membership test. |
| 6 | technical | Backward-compat regression at render: if `buildQACard` reads `q.recommended_choice` without an undefined guard, existing/pre-field questions (field absent) throw or badge spuriously — breaking Principle 4 (no-op when absent). | Render a fixture with no `recommended_choice`: chips must be byte-identical to today, no badge. Inspect the render reads the field defensively (`undefined`-safe). |
| 7 | technical | Contract-doc clobbering (epic pre-mortem risk #7): this story's additive `recommended_choice` row in `docs/headless-contract.md` §3 is dropped or overwritten by a sibling story's edit when the batch merges to the epic branch. | At epic integration, diff the `QAQuestion` field table — the `recommended_choice` row must coexist with sibling stories' additions; flag any wholesale rewrite of the table. |
